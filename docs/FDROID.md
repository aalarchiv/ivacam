# Publishing ivaCAM to F-Droid — Assessment & Draft Build Recipe

Status: **research / planning only**. Nothing here has been wired into the
build. This document is the assessment the maintainer asked for, plus a
draft `fdroiddata` metadata file and a draft Fastlane layout. Treat the
YAML and shell snippets as *first drafts to iterate on*, not tested CI.

Ground-truth facts about ivaCAM used below (verified against the repo at
the time of writing):

- App id `org.opcenter.ivacam`, `version` 0.3.0 in
  `crates/ivac-tauri/tauri.conf.json`. Tauri derives Android
  `versionName=0.3.0` / **`versionCode=3000`** (seen in the generated
  `gen/android/app/tauri.properties`; the gradle file reads those
  properties). License **GPL-3.0-or-later** (workspace `Cargo.toml`).
- Desktop+mobile shell is **Tauri 2** (`crates/ivac-tauri`); Android is
  built with `cargo tauri android build`. The Svelte UI loads from the
  embedded `dist/` in the system WebView and calls **native `ivac-core`
  over Tauri IPC** — *not* wasm (`docs/BUILDING.md` "Android (Tauri
  mobile)"; transport selection in `frontend/src/lib/api/client.ts` →
  `tauri.ts`). `ivac-core` is pure Rust, zero `*-sys`/C deps.
- Frontend: Svelte 5 + Vite, built with **pnpm** (`frontend/`).
  `pnpm build` runs `frontend/scripts/ensure-wasm-fresh.mjs` →
  **wasm-pack**. The wasm pkg (`crates/ivac-wasm/pkg`) is a pnpm `link:`
  dep used **only by the browser/static transport** (`wasm.ts` /
  `wasm.worker.ts` / `sim/driver.ts`) — the Android APK never runs it, but
  the pkg must still *exist* for `vite build` to resolve the import. The
  guard now builds it via `wasm-pack --mode no-install` (no build-time
  downloads) when `wasm-bindgen-cli` + `binaryen` are on `PATH`; see §2a.
- Remote `github.com/aalarchiv/ivacam`; releases git-tagged `v*`.
  `crates/ivac-tauri/gen/android/` is **git-ignored** (`.gitignore` →
  `crates/ivac-tauri/gen`) and regenerated by `cargo tauri android init`.

### Build metadata file
[Build Metadata Reference](https://f-droid.org/en/docs/Build_Metadata_Reference/)

- `metadata/org.opcenter.ivacam.yml` in a fork of
  [fdroiddata](https://gitlab.com/fdroid/fdroiddata). Fields:
  `Categories`, `License`, `SourceCode`, `IssueTracker`, `RepoType`/`Repo`,
  a `Builds:` list, `AutoUpdateMode`, `UpdateCheckMode`,
  `CurrentVersion`, `CurrentVersionCode`. See draft in §3.
- Each `Builds:` entry: `versionName`, `versionCode`,
  `commit` (use the **tag** or full hash, not a branch), plus the
  scripting fields `sudo` / `init` / `prebuild` / `build`, and
  `ndk`, `gradle`, `output`, `scandelete`/`scanignore`.

### Build-server reality
[Build Server Setup](https://f-droid.org/en/docs/Build_Server_Setup/),
[Building Applications](https://f-droid.org/en/docs/Building_Applications/)

- The app is built in an **isolated VM**. The documented phase split is:
  `init`/`prebuild` are for *preparing source* (this is where network
  fetches such as `cargo install`, `rustup target add`,
  `flutter pub get`, `pnpm fetch` happen in real Rust/Flutter recipes);
  `build` is for the *actual build* and runs under network isolation.
  **Net: anything that touches the network must happen in `sudo`/`prebuild`,
  never in `build`.** (Confirmed empirically by every Rust-on-F-Droid
  recipe — see the ruffle / gitsync precedents in §2, which do all
  `cargo install` / `rustup target add` in `sudo`/`prebuild`.)
- `sudo:` runs `sudo bash -x -c …` with full root, **state reset after
  each build** — used to `apt-get install` build deps (clang, cmake,
  rustup, etc.).
- **Scanner** rejects pre-compiled binaries / non-free blobs in the
  source tree. Any committed build artifacts (e.g. a checked-in
  `ivac-wasm/pkg/*.wasm`, vendored jars) must be excluded with
  `scanignore`/`scandelete` or not committed.

### Reproducible builds (optional)
[Reproducible Builds](https://f-droid.org/en/docs/Reproducible_Builds/) —
"Reproducible builds are **not a requirement**… we do consider their use
best practice." For Rust the docs recommend `SOURCE_DATE_EPOCH` and
`--remap-path-prefix`. We can ship F-Droid-signed first and pursue
reproducibility later.

## 2. Blocker analysis (specific to this stack)

### 2a. wasm-pack downloads at build time — SOLVED (offline build, wasm kept).

The scariest-looking blocker. Background:

- `pnpm build` → `ensure-wasm-fresh.mjs` → `wasm-pack build …`. By
  default wasm-pack **downloads `wasm-bindgen` and `wasm-opt` (binaryen)
  binaries** from GitHub on first run. On the F-Droid `build` phase there
  is no network, so a naive `cargo tauri android build` (whose
  `beforeBuildCommand` is `pnpm --dir ../frontend build`) **would fail
  offline**.
- The Android APK does not *run* the wasm pkg (it uses native `ivac-core`
  over Tauri IPC; wasm is the browser/static transport only). But the pkg
  must still **exist** for `vite build` to resolve `sim/driver.ts`'s
  un-`@vite-ignore`'d `import('ivac-wasm')` and the pnpm `link:` dep — so
  "just delete it" doesn't work; the build needs it present.

**Implemented fix — build wasm offline, don't skip it.**
`ensure-wasm-fresh.mjs` now runs **`wasm-pack build --mode no-install`**
whenever a matching `wasm-bindgen` CLI *and* a `wasm-opt` are on `PATH`
(it reads the required `wasm-bindgen` version from `Cargo.lock` and only
takes this path on an exact match; otherwise it falls back to the
downloading mode and says why). `--mode no-install` uses those PATH tools
instead of downloading anything — **verified to build the pkg with the
download cache removed and no network use**. wasm stays fully functional
everywhere; the build just stops reaching out.

For F-Droid this means: install the two tools in the **network-enabled
`prebuild`/`sudo` phase**, and the offline `build` phase then builds wasm
with zero downloads — no env-var gate, no skipping:

```
cargo install wasm-bindgen-cli --version <wasm-bindgen ver from Cargo.lock> --locked
apt-get install binaryen        # provides wasm-opt
```

(The same change also kills the flaky concurrent-download race that has
corrupted local builds, so it's a win beyond F-Droid.)

### 2b. pnpm dependency fetch must happen in `prebuild`, not `build`

Even without wasm, `pnpm install` needs the registry. Do it in the
network-enabled phase:

- `prebuild`: `corepack enable` (or `npm i -g pnpm@<pinned>`), then
  `pnpm --dir frontend install --frozen-lockfile` (lockfile committed —
  it is). Optionally `pnpm fetch` into a content-addressed store first.
- `build`: run the Tauri/gradle build with `pnpm` already populated and
  `--offline` where possible; `ensure-wasm-fresh` builds wasm offline via
  `--mode no-install` (tools installed in `prebuild` — see 2a).

Cargo deps are fetched implicitly the same way: do a `cargo fetch`
(workspace) in `prebuild` so the `build` phase compiles offline, or rely
on F-Droid's cargo handling. Pin everything (`Cargo.lock`, `pnpm-lock.yaml`
are both committed — good).

### 2c. Rust + NDK + Node/pnpm provisioning on the build server

Pattern, taken from real Rust-on-F-Droid recipes
([rs.ruffle.yml](https://gitlab.com/fdroid/fdroiddata/-/blob/master/metadata/rs.ruffle.yml),
[com.viscouspot.gitsync.yml](https://gitlab.com/fdroid/fdroiddata/-/blob/master/metadata/com.viscouspot.gitsync.yml)):

- **NDK**: set via the `ndk:` field (e.g. `27.2.12479018`, matching
  `docs/BUILDING.md`). F-Droid provisions that NDK; export
  `NDK_HOME`/`ANDROID_NDK_HOME` for cargo's Android linkers.
- **Rust**: `sudo: apt-get install -y rustup`, then in `prebuild`
  `rustup default <pin>` + `rustup target add aarch64-linux-android
  armv7-linux-androideabi i686-linux-android x86_64-linux-android`.
  (ruffle reads the channel from `rust-toolchain.toml`; we can pin a
  channel the same way or hardcode.)
- **Node/pnpm**: `sudo: apt-get install -y nodejs npm` (or use the F-Droid
  image's Node), then `corepack enable` / `npm i -g pnpm@<pin>` in
  prebuild.
- **JDK**: export `JAVA_HOME=/usr/lib/jvm/java-21-openjdk-amd64` (OpenJDK,
  satisfies the FLOSS-toolchain rule; Gradle needs JDK 21 per our docs).
- **Tauri CLI**: `cargo install tauri-cli --version ^2 --locked` in
  prebuild (the `cargo tauri android` driver).

No Tauri-2-Android app appears to be on F-Droid yet (searches surface
Tauri docs and Rust/Flutter apps, not a packaged Tauri mobile app). So
**this would likely be among the first Tauri-2 Android apps on F-Droid** —
expect reviewer back-and-forth and no copy-paste precedent for the
`cargo tauri android build` invocation specifically. The Rust-cross +
gradle mechanics are well-trodden (ruffle, gitsync); the Tauri wrapper is
the novel part.

### 2d. `gen/android` is git-ignored — needs a reproducible scaffold

F-Droid builds from a tag; the gradle project must exist at that commit.
Two options:

- **(A) Scaffold in `prebuild`** (keeps repo clean): run
  `cargo tauri android init` in prebuild to regenerate
  `crates/ivac-tauri/gen/android`. Pros: nothing extra committed, matches
  current dev flow. Cons: the scaffold is templated by the installed
  Tauri CLI version, so **pin `tauri-cli` exactly** or the generated
  gradle/manifest can drift between the tag and the build → non-reproducible
  and a moving target for reviewers.
- **(B) Commit `gen/android`** at release time: deterministic, reviewer
  can read the exact gradle/manifest, no CLI-version drift. Cons: large
  generated tree in-repo; must keep `.gradle`/`build`/`local.properties`
  ignored (the scaffold's own `.gitignore` already excludes those) and
  `scanignore` anything the scanner flags.

**Recommendation: (B) commit the scaffold for the released tag** (or a
trimmed, source-only subset), because F-Droid strongly favors a build
that reads from a fixed, inspectable tree over one that regenerates
project files from a network-installed tool. If we go (A), pin the Tauri
CLI version in the recipe and document it.

### 2e. Anti-features / non-free check

- **WebView**: Tauri Android uses the **system WebView** (Android System
  WebView / the OEM's). F-Droid's standard note here is the
  `NonFreeNet`/WebView caveat is generally *not* triggered just by using
  the platform WebView (it's a system component, not bundled). No
  Chromium is shipped. No GMS, Firebase, analytics, ads, or telemetry in
  ivaCAM. **Expected AntiFeatures: none.** (Double-check no crash
  reporter / update-checker phones home.)
- All native code is our own Rust + crates from crates.io (FOSS); no
  proprietary `*-sys`/vendored blobs (pure-Rust core). Audit transitive
  crate + npm licenses for GPL-compatibility before submitting.

### 2f. Reproducible-build feasibility

Feasible but not required. Rust + gradle reproducibility needs
`SOURCE_DATE_EPOCH` (from the commit date), `--remap-path-prefix` for
cargo registry + repo paths (`CARGO_ENCODED_RUSTFLAGS`, as gitsync does),
deterministic gradle (`-Pandroid...`), and stripping the wasm step
(option 2a-1 also helps determinism). **Plan: ship F-Droid-signed first;
chase developer-signature-verified reproducibility as a follow-up.**

## 3. Draft `fdroiddata` metadata — `metadata/org.opcenter.ivacam.yml`

> Draft. `versionCode` 3000 and tag `v0.3.0` are real; the `commit:` must
> be the **release tag** (or its full hash). The shell blocks are a
> first pass and *will* need iteration with a real build on the F-Droid
> buildserver (esp. the `cargo tauri android build` output path and the
> pnpm/cargo offline handling).

```yaml
Categories:
  - Graphics
  - Science & Education
License: GPL-3.0-or-later
AuthorName: Soenke J. Peters
SourceCode: https://github.com/aalarchiv/ivacam
IssueTracker: https://github.com/aalarchiv/ivacam/issues
# Changelog: https://github.com/aalarchiv/ivacam/releases

RepoType: git
Repo: https://github.com/aalarchiv/ivacam.git

Builds:
  - versionName: 0.3.0
    versionCode: 3000
    commit: v0.3.0
    subdir: crates/ivac-tauri
    sudo:
      - apt-get update
      - apt-get install -y rustup nodejs npm clang cmake make pkg-config
    ndk: 27.2.12479018
    prebuild:
      # --- toolchains (network IS available in prebuild) ---
      - export JAVA_HOME=/usr/lib/jvm/java-21-openjdk-amd64
      - rustup default 1.88.0   # pin to whatever ivaCAM builds with
      - rustup target add aarch64-linux-android armv7-linux-androideabi i686-linux-android x86_64-linux-android
      - cargo install tauri-cli --version '^2' --locked
      - corepack enable || npm i -g pnpm@9
      # --- wasm toolchain so the offline build skips downloads (see 2a). The
      #     CLI version MUST match the wasm-bindgen crate in Cargo.lock. ---
      - cargo install wasm-bindgen-cli --version 0.2.120 --locked
      - apt-get install -y binaryen   # provides wasm-opt
      # --- frontend deps ---
      - pnpm --dir ../../frontend install --frozen-lockfile
      # --- cargo deps prefetched so the build phase is offline-clean ---
      - cargo fetch --manifest-path ../../Cargo.toml
      # --- scaffold the android gradle project (if not committed) ---
      - cargo tauri android init --skip-targets-install
    build:
      - export JAVA_HOME=/usr/lib/jvm/java-21-openjdk-amd64
      - export NDK_HOME=$ANDROID_NDK_HOME
      # ensure-wasm-fresh auto-selects `wasm-pack --mode no-install` because the
      # wasm-bindgen CLI + binaryen were installed in prebuild — no downloads.
      - export SOURCE_DATE_EPOCH=$(git -C ../.. log -1 --pretty=%ct)
      - cargo tauri android build --apk --split-per-abi --target aarch64 armv7 i686 x86_64
    # Per-ABI unsigned release APKs land here:
    output: gen/android/app/build/outputs/apk/universal/release/app-universal-release-unsigned.apk
    scandelete:
      # generated/build artifacts the scanner should not police
      - gen/android/.gradle
      - gen/android/app/build

AutoUpdateMode: Version v%v
UpdateCheckMode: Tags ^v[0-9.]+$
CurrentVersion: 0.3.0
CurrentVersionCode: 3000
```

Notes / known-unknowns to resolve with a real build:

- **`output` path is uncertain.** Verified (debug) tree from our own
  `--split-per-abi` build: Tauri names the gradle flavors by **Rust target
  short name**, so the APKs are
  `gen/android/app/build/outputs/apk/<flavor>/debug/app-<flavor>-debug.apk`
  for `flavor ∈ {arm64, arm, x86, x86_64}` (NOT the `arm64-v8a` Android
  ABI names). Release mode should mirror that as
  `.../<flavor>/release/app-<flavor>-release-unsigned.apk`. F-Droid wants
  one Builds entry per versionCode, so multi-ABI means either distinct
  per-ABI versionCodes or a single universal APK
  (drop `--split-per-abi` → `apk/universal/release/...`). Decide, then
  confirm the exact release path with a real build (we only built `--debug`).
- **Release vs debug + signing.** F-Droid wants the **unsigned release**
  APK and signs it itself; ensure the gradle release variant is not
  configured to require our keystore (`gen/android/app/build.gradle.kts`
  pulls `keystore.properties` — must be absent/optional in the F-Droid
  build so the release variant produces an unsigned APK).
- If we **commit** `gen/android` (option 2d-B), drop the
  `cargo tauri android init` line and keep the `scandelete` for build
  dirs.
- `VercodeOperation` is not needed (Tauri's 0.3.0→3000 mapping is already
  the real versionCode).

## 4. Repo changes needed (prioritized)

Legend: **[BLOCKER]** = submission can't proceed cleanly without it;
**[should]** = strongly expected by reviewers; **[nice]** = polish.

1. **[DONE] Offline wasm build.** `frontend/scripts/ensure-wasm-fresh.mjs`
   now runs `wasm-pack build --mode no-install` when a matching
   `wasm-bindgen` CLI + `wasm-opt` are on `PATH`, so the build downloads
   nothing (verified with the cache removed). For F-Droid, install
   `wasm-bindgen-cli` (pinned to the `Cargo.lock` version) + `binaryen` in
   the network-enabled `prebuild` phase; the offline `build` phase then
   builds wasm with no network. wasm is kept, not skipped. (See §2a.)

2. **[BLOCKER] Add Fastlane metadata to ivaCAM's repo** under
   `fastlane/metadata/android/en-US/`:
   - `title.txt` → `ivaCAM`
   - `short_description.txt` (≤80 chars, no trailing period), e.g.
     `DXF/SVG to G-code CAM for CNC mills, lasers and drag-knife plotters`
   - `full_description.txt` (longer marketing/feature text)
   - `images/icon.png` (reuse `crates/ivac-tauri/icons/icon.png` /
     `android/mipmap-xxxhdpi`)
   - `images/phoneScreenshots/1.png` (+ more) — **need real screenshots
     of the Android app**; currently none exist.
   - `changelogs/3000.txt` (≤500 chars) — the 0.3.0 notes.
   Currently the repo has **no** `fastlane/` tree at all.

3. **[BLOCKER] Decide & implement the `gen/android` strategy** (commit the
   scaffold for the tag, or pin `tauri-cli` and scaffold in `prebuild`).
   Recommended: commit a source-only scaffold at release time, keep
   `.gradle`/`build`/`local.properties`/keystore files ignored. Ensure
   the release variant produces an **unsigned** APK (no mandatory
   keystore). (See §2d.)

4. **[should] Dependency license audit.** Run `cargo-deny` (licenses) over
   the workspace and check npm dep licenses for GPL-3.0-compatibility.
   Document results; fix or replace anything incompatible. (`scripts/`
   already references `cargo-deny` as an optional gate.)

5. **[should] Pin/confirm toolchain versions** the recipe hardcodes:
   Rust channel (add a `rust-toolchain.toml` so the recipe can read it
   instead of hardcoding), NDK `27.2.12479018`, pnpm major, `tauri-cli`
   `^2`. Lockfiles (`Cargo.lock`, `pnpm-lock.yaml`) are already committed
   — keep them current at the tag.

6. **[should] versionCode strategy doc.** Tauri maps `x.y.z` →
   `versionCode` deterministically (0.3.0 → 3000). Document the mapping
   so each tagged release yields a strictly increasing, unique
   versionCode; F-Droid `UpdateCheckMode: Tags` will read it from the
   tagged gradle/manifest. Watch the ceiling (z and y are 2-digit fields
   in Tauri's scheme).

7. **[nice] Reproducible-build hardening**: `SOURCE_DATE_EPOCH`,
   `CARGO_ENCODED_RUSTFLAGS` remap-path-prefix, deterministic gradle.
   Pursue after first inclusion. (See §2f.)

8. **[nice] Fork fdroiddata, add `metadata/org.opcenter.ivacam.yml`**
   (§3), run `fdroid lint org.opcenter.ivacam` /
   `fdroid build org.opcenter.ivacam` locally, open the RFP issue +
   merge request. This is the actual submission, gated on 1–3.
